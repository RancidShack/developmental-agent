**Author:** Nicholas P M Baker, Synapstak Ltd
**Date:** 1 May 2026
**Repository:** github.com/RancidShack/developmental-agent
**Status:** Pre-registration — committed before any v1.2 code is written

---

## 1. Programme context and what v1.2 is designed to test

The v1.1 iteration (Baker, 2026r, 2026s) established that the architecture's flag structures can be transformed from binary states into records of their own formation, through a parallel observer module that introduces no behavioural modification. The v1.1 batch confirmed byte-identical preservation against the v0.14 and v1.0 baselines; the Category γ finding demonstrated that the records carry agent-individual structural variation largely orthogonal to experimental conditions. The architecture under v1.1 holds formation records for four flag types — threat, mastery, knowledge-banking, end-state activation and banking — with bidirectional cross-references between threat flags and the knowledge-banking flags that derive from the same coordinate through v0.14 competency-gated transformation.

What the v1.1 architecture does not yet have — and what v1.2 is designed to provide — is an explicit representation of the categorical structure within which those records sit. The agent under v1.0 / v1.1 uses its categories implicitly: its primitive bias mechanism distinguishes ATTRACTOR cells from KNOWLEDGE cells from FRAME cells; its phase-transition logic distinguishes Phase 1 from Phase 2 from Phase 3; its mastery and threat layers distinguish which flag types operate on which cells. None of these distinctions are represented in the agent as a structure the agent can examine. The agent acts on its schema but cannot interrogate it.

v1.2 operationalises SICC Commitment 1 (schema as embodied a priori): the agent acquires an explicit, agent-readable representation of the categories of its environment — cell types as kinds, actions as available verbs, phases as developmental periods, flag types as formation categories. The schema is given at agent construction time, not learned. The agent does not yet act on the schema; the schema does not yet drive the action-selection pipeline. What v1.2 contributes is making the schema *visible* to the agent — held as an inspectable structure — as the substrate for subsequent iterations (layered properties, reporting) that will make the schema consequential.

The Montessori framing, established in the v1.1 paper (Baker, 2026r, Section 1.2), is load-bearing for the trajectory. The prepared environment is the same across runs at matched parameters; what the agent carries away includes not only the route it traversed (the v1.1 formation records) but the categories within which the traversal was possible — the kinds of things it encountered, the moves available to it, the periods through which its development was organised. v1.2 gives the agent an explicit representation of those categories.

## 2. Architectural specification

### 2.1 What the schema is

The schema is a structured object the agent holds from construction time and can examine on demand. It is not learned; it is given. It does not drive behaviour; it is not consulted by the action-selection pipeline, the drive composition, the value function, or the model updates. The schema is a read-accessible architectural object.

The schema comprises four sub-schemas, each covering a distinct categorical domain of the agent's operation:

**The cell-type schema** enumerates the cell types the agent's environment contains (FRAME, NEUTRAL, HAZARD, ATTRACTOR, END_STATE, KNOWLEDGE) with their architectural properties represented as named fields: passability (boolean), cost_on_entry (scalar or null), feature_reward_eligible (boolean), attraction_bias_eligible (boolean), gating_eligible (boolean — whether the agent's hard-gate action-selection rule applies to cells of this type), transformation_eligible (boolean — whether cells of this type can undergo v0.14 competency-gated transformation). The property values are inherited from the v0.14 / v0.13 / v0.10 architecture; the schema makes them explicit in a form the agent can examine by cell type.

**The action schema** enumerates the four actions available to the agent (UP, DOWN, LEFT, RIGHT) with their directional effects on agent position as (dx, dy) offset pairs. The schema makes the action vocabulary explicit as a structure rather than as implicit branches in the action-selection pipeline.

**The phase schema** enumerates the three developmental phases (PHASE_1, PHASE_2, PHASE_3) with the following fields per phase: phase_name (string), drive_composition (a list of active drive names), transition_condition (a string description of what triggers entry to this phase — path_completion for Phase 1, proportional_run_length for Phase 2/3), drive_weights (a dict of drive names to weight scalars, null for drives not active in this phase). The schema makes the developmental schedule explicit as a structure the agent holds rather than a schedule encoded only in the batch-runner's phase-transition logic.

**The flag-type schema** enumerates the four flag types from v1.1 (THREAT, MASTERY, KNOWLEDGE_BANKING, END_STATE) with the following fields per flag type: formation_condition (string — what event causes a flag of this type to be set), formation_threshold (integer or null — number of qualifying events required, where applicable), confirming_observation_operationalisation (string — per the v1.1 pre-registration §2.2 specification), disconfirming_observation_semantics (string — the narrow semantics per v1.1 §2.3), cell_types_applicable (list of cell type names — which cell types the flag type operates on). The schema makes the v1.1 record structure explicit at the type level, providing the categorical substrate within which the v1.1 records' formation events are situated.

### 2.2 Architectural choices settled here

**Per-instance, not per-class.** The schema is instantiated per agent at construction time, not defined at the agent-class level. The carry-forward's provisional lean was per-instance, on the principle that subsequent iterations will introduce per-agent variation (in the reporting phase, an agent's schema may be queried against the agent's own records, and the query results will vary per agent even with identical schemas). Instantiating per-agent ensures that schema objects are agent-bound from the start, without requiring a re-architecture when per-agent variation is introduced.

**Separate schema file, not inlined in per-run CSV.** The v1.2 instrumentation writes the schema for each run to a separate `schema_v1_2.csv` (one row per agent per run), not inlined in the per-run metrics CSV. The schema content is largely identical across runs at matched parameters (the cell types, actions, phases, and flag types are determined by the architecture, not by the seed); inlining would inflate the per-run CSV with largely duplicated content. The separate CSV permits cross-run schema verification (Category β) while keeping the per-run metrics CSV format stable.

**Agent-side read capability, read-only and without behavioural consequence.** The v1.2 agent exposes a `query_schema(domain, key=None)` method that returns schema content for a named domain ('cell_types', 'actions', 'phases', 'flag_types') or for a specific entry within a domain. The method is callable within the batch runner for verification and logging purposes; it is not called during the action-selection pipeline, drive computation, model updates, or any other behavioural pathway. This is parallel to v1.1's design boundary — in v1.1, the records exist but the agent does not act on them; in v1.2, the schema exists and the agent can read it, but the reading does not yet drive anything. The reporting iteration is where the agent reads schema to produce reports.

**SICC commitments are not included in the schema as a meta-layer.** The carry-forward's provisional lean was no; this is confirmed here. A meta-layer representing the SICC commitments themselves would operationalise a much later phase of the trajectory — one associated with the substrate-transposition phase where the cognitive layer needs self-awareness sufficient to migrate. Including it here would pre-empt work reserved for later and would compromise the single-architectural-change discipline.

### 2.3 Implementation surface

The v1.2 implementation surface is more substantial than v1.1's but the architectural change remains contained at the research-question level. The implementation modifies:

- `curiosity_agent_v1_2.py`: The agent class acquires a `schema` attribute instantiated at `__init__` via a `_build_schema()` method. The `_build_schema()` method constructs the four sub-schemas from the agent's and world's existing constants and specifications. The `query_schema()` method provides read access. No existing method is modified; the schema is an additive structure.

- `v1_2_schema_observer.py`: A parallel observer module (modelled on `v1_1_provenance.py`) that reads the agent's schema at end-of-run and writes the schema fields to `schema_v1_2.csv`. The module has read-only access to the agent's schema.

- `curiosity_agent_v1_2_batch.py`: The batch runner, which imports `curiosity_agent_v1_2.py` and runs the v1.0 recorder, v1.1 provenance store, and v1.2 schema observer as parallel observers.

- `verify_v1_2_disabled.py`: Level-1 pre-flight verification (against v0.14 baseline with all instrumentation disabled).

- `verify_v1_2_no_schema.py`: Level-2 pre-flight verification and permanent regression test (against v1.1 baseline with v1.1 provenance enabled, v1.2 schema observer disabled).

### 2.4 What v1.2 does not change

All architectural elements from v0.14 and v1.1 are preserved unchanged. The agent's action-selection pipeline, drive composition, value learning, model updates, phase-transition logic, and primitive bias mechanism are unmodified. The v1.1 provenance store operates unchanged. The schema does not modify any behavioural pathway; it is an additive structure held alongside the agent's behavioural machinery, not wired into it. The parallel-observer pattern from v1.1 is maintained; the v1.2 schema observer is an additional parallel observer, not a modification of existing observers.

## 3. Experimental matrix

The experimental matrix matches the v1.1 batch: one architecture (v1.2) crossed with six cost levels (0.1, 0.5, 1.0, 2.0, 5.0, 10.0) crossed with three run lengths (20,000, 80,000, 160,000 steps) crossed with ten runs per cell, totalling 180 runs.

For each (cost, run length, run index) triple, the random seed is matched to the v1.1 batch (Baker, 2026r), which itself matched v1.0, v0.14, v0.13, v0.12, v0.11.2, and v0.10 at the same conditions. Seeds are loaded from `run_data_v1_1.csv` rather than regenerated, on the methodological principle (established in the v0.12 pre-registration and held through every subsequent iteration) that Python's hash randomisation per session prevents deterministic seed regeneration; the recorded seeds in the CSV are the data of record.

Under matched seeds and the parallel-observer implementation pattern — in which the agent's behaviour is not modified — the v1.2 batch is expected to reproduce v1.1 per-run metrics byte-for-byte. This prediction is the pre-flight verification commitment; it is tested before the batch rather than asserted after it.

## 4. Pre-flight verification commitments

Two pre-flight verifications are committed here and must both pass before the v1.2 batch is run.

**Level 1 (`verify_v1_2_disabled.py`):** The v1.2 batch runner with all instrumentation disabled (`--no-instrument --no-provenance --no-schema`) produces output bit-for-bit identical to v0.14 baseline at matched seeds, across 10 matched-seed runs at cost 1.0, 20,000 steps. This confirms that neither the v1.0 recorder, the v1.1 provenance store, nor the v1.2 schema observer — individually or jointly — interfere with the agent's RNG state or computation when not instantiated.

**Level 2 (`verify_v1_2_no_schema.py`):** The v1.2 batch runner with v1.0 instrumentation and v1.1 provenance enabled but v1.2 schema observer disabled (`--no-schema`) produces output bit-for-bit identical to v1.1 baseline at matched seeds, across the same 10 matched-seed runs. This is the permanent regression test added to the verification pipeline; it will catch any future coupling of prior instrumentation layers to v1.2 schema state. The v1.1 level-2 verification (`verify_v1_1_no_provenance.py`) remains in the pipeline; v1.2 adds to it rather than replacing it.

Both verifications must pass at 10 of 10 runs before the v1.2 batch is run. A failure in either verification requires investigation and resolution before proceeding; the batch does not run until verification is clean.

## 5. Interpretation categories

Six interpretation categories are pre-registered for v1.2. Category Φ and Category Ω are the honesty constraint and architectural-statement claim respectively, as established in the v1.1 pre-registration framework.

### Category α: Behavioural preservation

**Criterion:** The v1.2 architecture introduces no behavioural modification. Under matched seeds, v1.2 per-run metrics match v1.1 baseline values for every v1.1-visible metric field.

**Operationalisation:** The level-2 pre-flight verification passes at 10 of 10 runs before the batch. Following the batch, internal consistency between v1.1 metric fields and v1.2 outputs holds at 180 of 180 runs across all v1.1-visible couplings: `attractors_mastered`, `hazards_flagged`, `hazards_banked_as_knowledge`, `activation_step` presence, `end_state_banked` status, and the v1.1 provenance count couplings, all match v1.1 baseline values at matched seeds.

**Threshold:** Category α passes if and only if (a) both pre-flight verifications pass at 10/10, and (b) all v1.1-visible metric fields match v1.1 baseline at 180 of 180 runs. Partial pass is not defined for Category α; the parallel-observer implementation pattern either preserves behaviour completely or it does not.

**Note on the time-as-window principle:** Category α's preservation check applies to v1.1-visible fields. Fields that v1.2 adds (schema content fields) are outside the scope of v1.1-baseline comparison by definition. Category β covers those.

### Category β: Schema completeness

**Criterion:** The schema the v1.2 architecture introduces contains every category the v0.14 / v1.1 architecture actually uses.

**Operationalisation:** Four sub-checks, one per schema domain:

- *β1 (cell-type schema):* Every cell type code that appears in the v0.14 world construction (`FRAME`, `NEUTRAL`, `HAZARD`, `ATTRACTOR`, `END_STATE`, `KNOWLEDGE`) has a corresponding entry in the agent's cell-type schema. All six types appear in `schema_v1_2.csv` for every run; no run records a cell type encountered in the world that is absent from the schema.

- *β2 (action schema):* All four action codes used in the v0.14 action-selection pipeline (`UP`, `DOWN`, `LEFT`, `RIGHT`) have entries in the action schema with correct (dx, dy) offset values. Verified by comparing the schema's offset values against the agent's actual movement outcomes at matched seeds in the level-2 verification run.

- *β3 (phase schema):* All three phases the agent passes through (Phase 1 prescribed acquisition, Phase 2 drive-based integration, Phase 3 preference-weighted autonomy) have entries in the phase schema with drive-composition and transition-condition fields populated consistently with the v0.14 batch-runner's phase-transition logic.

- *β4 (flag-type schema):* All four flag types recorded in the v1.1 provenance store (`THREAT`, `MASTERY`, `KNOWLEDGE_BANKING`, `END_STATE`) have entries in the flag-type schema, with formation_condition, confirming_observation_operationalisation, and cell_types_applicable fields consistent with the v1.1 pre-registration §2.2 and §2.3 specifications.

**Threshold:** Category β passes if all four sub-checks pass at 180 of 180 runs. A sub-check failure (e.g. a cell type present in the world but absent from the schema) is a Category D signal, not a partial-pass finding.

### Category γ: Schema–record alignment

**Criterion:** The v1.2 schema and the v1.1 records are structurally consistent at the type level. The flag-type schema's properties correctly describe the records the v1.1 provenance store produces.

**Operationalisation:** Three alignment checks:

- *γ1 (formation condition consistency):* The `formation_condition` field in the flag-type schema, for each flag type, matches the conditions under which flags of that type form in the v1.1 records. Operationally: for THREAT flags, the schema's `formation_condition` field specifies the v0.10 three-entry conversion rule and the v0.12 signature-matching first-entry rule; the v1.1 records' `flag_set_step` values are consistent with one of these two mechanisms for every threat flag in the batch.

- *γ2 (cross-reference structural consistency):* The cell-type schema's `transformation_eligible` field is `True` for `HAZARD` and `False` for all other types. The v1.1 records' bidirectional cross-references (threat-flag to knowledge-banking-flag pairs) occur only on cells whose type at formation was `HAZARD`. Verified by cross-referencing the schema's `transformation_eligible` flags against the v1.1 provenance store's cross-reference records.

- *γ3 (phase-composition consistency):* The phase schema's `drive_composition` field for each phase correctly enumerates the drives active in that phase as specified in the v0.14 batch runner. No drive appears in the schema as active in a phase where the batch runner does not activate it; no drive active in the batch runner is absent from the schema.

**Threshold:** Category γ passes at 180 of 180 runs for γ1 and γ3 (which are schema-level rather than per-run checks, and should hold uniformly). γ2 is verified against the v1.1 batch's cross-reference records directly; it passes if zero cross-references occur on cells of non-transformation-eligible types.

**Note on the time-as-window principle:** Category γ does not require the agent to have encountered every schema-defined category within any given run. A KNOWLEDGE cell absent from a run's v1.1 records does not constitute a γ-failure; the schema is assessed against what it *can* hold, not what has been populated within the run-length window. The schema is complete if every category it defines is correctly specified; it is not complete only if a category is specified incorrectly or omitted.

### Category δ: Negative findings honestly recorded

Three categories of negative finding are anticipated and committed to honest reporting in advance.

**δ1 (schema sparsity for short runs):** At the 20,000-step run length, not all cell types will have been encountered in every run — in particular, KNOWLEDGE cells may not form in runs where competency thresholds have not been reached within the run-length window. The schema's completeness is not a function of what was encountered; the schema holds the entry regardless. δ1 records, per run length, which schema entries correspond to categories that appeared in the run's v1.1 records versus categories defined in the schema but not instantiated within the run's observation window.

**δ2 (agent read-capability without behavioural consequence):** The `query_schema()` method exists and returns correct content. The agent does not call it during any behavioural pathway. This is confirmed via the level-2 pre-flight verification (byte-identical preservation with schema enabled) and logged explicitly in the δ2 finding as a commitment that the read capability is held without being used. The finding is not a failure; it is the pre-registered expected outcome at this iteration. The reporting iteration is where the read capability becomes consequential.

**δ3 (schema uniformity across the batch):** Because the schema is determined by the architecture rather than by the seed, the schema content should be identical across all 180 runs. δ3 records the uniformity rate as a check on implementation correctness. Any deviation from 180-of-180 uniformity in the schema content fields (excluding fields that are legitimately run-specific, if any are introduced) is an implementation error to be investigated, not a finding to be reported as substantive.

### Category Φ: Honesty constraint

The schema v1.2 introduces is the substrate for an eventual agent-side interrogation capability. Whether the agent can produce useful queries against the schema — and what counts as useful — is the question subsequent iterations answer, not a question the v1.2 batch can address.

The substantive reading of v1.2's contribution is that the schema gives the agent an explicit representation of the categories within which its formation records (v1.1) and future layered properties (v1.3) are situated; together, the records and the schema constitute the substrate that the reporting iteration (v1.4) will read to produce auditable self-accounts. The deflationary reading is that the schema is a structured lookup table attached to the agent, consulted by no behavioural pathway and producing no output the agent itself authors; its adequacy as a substrate for later iterations is asserted rather than demonstrated.

Both readings are consistent with the v1.2 batch. The v1.4 reporting iteration is the methodological vehicle for distinguishing them. The architectural claim made in any v1.2 paper is bounded by Category Φ accordingly.

### Category Ω: Architectural-statement claim

The architectural-statement claim for v1.2, subject to revision by a v1.2.x amendment if data-dependent operationalisation is required: the agent under v1.2 holds two complementary representations of its learning — the formation records (v1.1), which capture *what* the agent learned and *when*, and the schema (v1.2), which captures *within what categorical structure* the learning occurred. Together, these two representations constitute the substrate the reporting iteration needs to produce auditable self-accounts: records without categories cannot be narrated; categories without records are abstract structures with no instances. v1.2 provides the categorical component; v1.1 has already provided the record component. The two together are the substrate that v1.3 (layered properties) and v1.4 (reporting) build on.

This claim is bounded by Category Φ. Its verification rests on v1.4's output.

## 6. Amendment policy

The amendment budget for v1.2 is three amendments, following the convention established in v0.13 and held through v1.1.

Amendments are committed to the repository before any reanalysis is performed. Amendments are operational rather than architectural where possible: clarifying analysis operationalisations (as v1.1.1 and v0.13.1 did), not modifying the architecture's code or specification. An amendment that modifies `curiosity_agent_v1_2.py` or `v1_2_schema_observer.py` counts against the budget regardless of scope; an amendment that clarifies interpretation criteria or analysis scripts without touching the architectural code is still counted but noted as operational.

One amendment is anticipated as likely: a v1.2.1 amendment specifying the Category γ analysis operationalisation in the same manner as v1.1.1 specified the Category γ structural-distance metric. The specific operationalisation of γ1 (formation-condition consistency) requires examining the v1.1 batch's flag_set_step distribution against the schema's formation_condition fields; the precise test statistic to be computed should be committed after the schema exists to be examined but before reanalysis. This amendment is not pre-committed here because committing it now, before the records and schema exist, would risk specifying an operationalisation that the data structure does not support in the form specified. The lesson of v1.1's partial-pass outcome (Section 4 of Baker, 2026r) applies: pre-registered thresholds that depend on assumed data structure should be calibrated against the actual structure.

The remaining two amendments are reserved. If no further amendment is needed, v1.2 closes with unused budget; this is the disciplined outcome.

## 7. Mechanistic predictions

The pre-registration commits the following mechanistic predictions before any v1.2 code is written, as the basis against which the iteration's findings will be evaluated.

**Prediction 1 (Category α).** The parallel-observer implementation pattern produces byte-identical preservation by construction. v1.2's schema does not modify the agent or the world; the agent's behaviour under v1.2 is structurally guaranteed to match v1.1 baseline. Both pre-flight verifications will pass at 10 of 10.

**Prediction 2 (Category β).** The schema will be complete at 180 of 180 runs, because the cell types, actions, phases, and flag types are determined by the architecture and are enumerable in advance. Schema completeness is a property of the implementation's faithfulness to the specification, not a property of any particular run's trajectory. The check is essentially an implementation correctness check.

**Prediction 3 (Category γ).** The schema–record alignment checks will pass across the batch. The flag-type schema's formation conditions are derivable from the v0.10, v0.11.2, v0.12, and v0.14 specifications already documented in prior pre-registrations; the cell-type schema's transformation_eligible assignments follow directly from the v0.14 specification. The alignment is structural rather than empirical.

**Prediction 4 (Category δ, δ1).** Schema entries for KNOWLEDGE cells will correspond to categories instantiated in fewer runs at 20,000 steps than at 80,000 and 160,000, because knowledge-cell formation requires competency thresholds to be reached and the agent's competency builds over time. The number of runs with at least one knowledge-banking flag in the v1.1 batch was 19 of 60 at 20,000 steps, 51 of 60 at 80,000, and 59 of 60 at 160,000 (Baker, 2026r, Table 1). The δ1 finding will show KNOWLEDGE cell schema entries instantiated at comparable rates.

**Prediction 5 (Category δ, δ2).** The `query_schema()` method will confirm correct content return across all four domains. No behavioural pathway will call it; the level-2 pre-flight verification will confirm this at byte-identical resolution.

## 8. Relation to the SICC trajectory

The v1.2 iteration operationalises SICC Commitment 1 (schema as embodied a priori). The SICC document specifies this commitment as: the agent holds, from construction, a representation of the categorical structure of what it can know about. The schema is not the agent's knowledge; it is the structure within which knowledge is organised. The agent is not built to learn its schema; the schema is architectural rather than acquired. This is the a priori character of the commitment — not a priori in the Kantian sense of necessary truths, but in the sense of structural preconditions given before learning begins.

The v1.2 schema operationalises this commitment at the level of the agent's current environment: cell types as the kinds of things the environment contains; actions as the verbs available to the agent; phases as the periods through which development is organised; flag types as the categories of what is known. This is the minimal schema consistent with the architecture as it now stands.

SICC Commitment 9 (earned extensibility) specifies that the schema should, in principle, be capable of extension when new categories are earned through competency. This commitment is reserved for a later iteration. The v1.2 schema is fixed at construction time; extensibility is not operationalised here. The schema design is, however, consistent with future extensibility: per-instance instantiation ensures that schema objects are agent-bound; the `query_schema()` interface returns structured data that a future `extend_schema()` method could populate. The extension pathway is not closed by v1.2; it is simply not opened.

SICC Commitment 6 (layered property structure), Commitment 11 (reporting), and Commitment 10 (substrate transposition) are reserved for v1.3, v1.4, and v1.5 onwards respectively. Each is its own iteration with its own pre-registration.

## 9. Methodological commitments

The methodological discipline that has held throughout the programme holds for v1.2 without modification.

**Pre-registration before code.** This document is committed to the public repository at github.com/RancidShack/developmental-agent before any v1.2 code is written. The commit timestamp is the data of record for this commitment.

**Matched-seed comparison.** Seeds are loaded from `run_data_v1_1.csv` rather than regenerated. The seed chain links v0.10, v0.11.2, v0.12, v0.13, v0.14, v1.0, v1.1, and v1.2 at every (cost, run length, run index) cell.

**Parallel-observer pattern.** The v1.2 schema is implemented as a parallel observer module. The agent and world are not modified. The byte-identical preservation property is structurally guaranteed rather than aspirational.

**Accumulating regression test pipeline.** The v1.1 level-2 verification (`verify_v1_1_no_provenance.py`) remains in the pipeline. v1.2 adds `verify_v1_2_no_schema.py`. Both must pass before the v1.2 batch runs. The pipeline accumulates; it does not replace.

**Honest reporting of negative findings.** Category δ commits to three anticipated negative findings in advance. If additional unexpected negative findings arise, they are reported alongside the pre-registered categories without post-hoc suppression.

**Amendment budget.** Three amendments are available. Amendments are committed before reanalysis. The amendment budget at the close of v1.2 is reported in the v1.2 paper as a methodological record.

**Single-architectural-change discipline.** v1.2 introduces explicit schema as the singular architectural extension. All prior architectural elements are preserved unchanged. The discipline applies at the research-question level: what v1.2 asks is whether the architecture can hold an explicit categorical representation of its own environment, not learned, not behaviourally consequential, as the substrate for subsequent iterations to build on.

---

## 10. Stopping rule

If the level-1 or level-2 pre-flight verification fails, the batch does not run. The failure is investigated and resolved before proceeding. If resolution requires a code change to `curiosity_agent_v1_2.py` or `v1_2_schema_observer.py`, the change is committed as a v1.2.1 amendment (consuming one of the three amendment budget) and both verifications are re-run before the batch proceeds. A second verification failure consuming a second amendment would leave one amendment available for data-dependent operationalisation; if a third failure occurs, the iteration is suspended and the situation is recorded honestly in the pre-registration record as a methodological finding.
