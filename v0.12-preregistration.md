# v0.12 Pre-Registration: Category-Level Generalisation in the Threat Layer via Cell-Type Signature-Matching

**Author:** Nicholas P M Baker, Synapstak Ltd
**Date:** 25 April 2026
**Status:** Pre-registration, committed before any v0.12 code is written
**Inheritance baseline:** v0.11.2 (Baker, 2026g)
**Preceding work:** v0.11.2 amendment (Baker, 2026g), v0.11.1 amendment (Baker, 2026f), v0.11 pre-registration (Baker, 2026e), v0.10 (Baker, 2026d)

## 1. Reason for iteration

The v0.10 paper (Baker, 2026d) reported full stabilisation of Phase 3 hazard avoidance at 160,000 steps under the addition of an experience-driven persistent threat layer, with one boundary condition characterised honestly: at cost 5.0 at 20,000 steps, v0.10 produced 5 of 10 Phase 3 cleanness against the otherwise-uniform 8–10 of 10 across the matrix. The mechanism was identified in Section 4 of that paper and is uncontested. The per-cell-no-generalisation property of the v0.10 threat layer requires three entries per hazard cell to convert each cell's flag. Agents who encounter a novel hazard late in Phase 2 or during Phase 3 itself cannot complete the three-entry conversion within Phase 3's time budget when the run length is short and the cost is high enough for Phase 3 entries to count as violations. The boundary effect disappears at longer run lengths because Phase 3 has more steps available for late-encountered hazards to reach conversion.

The v0.10 paper named the architectural question this finding opens: would adding category-level generalisation to the threat layer eliminate the boundary condition by allowing late-encountered hazards to be flagged on first entry rather than after three? v0.10 reserved this as a separate research question for a subsequent iteration. The v0.12 iteration is that follow-on work.

The v0.12 iteration is therefore directed at a known, characterised, mechanistically-explained finding from v0.10. The architectural goal is identified: enable categorical generalisation in the threat layer so that late-encountered hazards can be flagged on first entry when the agent already has flagged at least one hazard of the same category. The behavioural target is identified: full elimination of the cost-5.0-at-20,000-steps outlier, with v0.10's full-stabilisation pattern at 160,000 steps preserved and v0.11.2's mastery findings preserved.

The v0.13 iteration (end-state target activation) is the more architecturally novel question and has fresher v0.11.2 momentum behind it. The decision to do v0.12 first is deliberate: the v0.10 outlier is the older finding, the threat-layer programme has a coherent arc (v0.9 → v0.10 → v0.12), and v0.13 can run after v0.12 with full v0.11.2 + v0.12 inheritance. The ordering choice is recorded here so that it is on the methodological record as deliberate rather than default.

## 2. The architectural change

The v0.12 architecture inherits v0.11.2 unchanged except for one modification to the flag-conversion rule in the threat layer.

Under v0.11.2 (and v0.10 before it), when the agent enters a hazard cell, the cell's per-cell entry counter increments by 1. When the counter reaches the pre-registered threshold of three, the cell's threat flag is set to 1. The conversion is one-way; flags do not clear. Each hazard cell is learned independently, with no mechanism for transferring flag status across cells.

Under v0.12, when the agent enters a hazard cell, the architecture first checks whether any other hazard cell has threat flag = 1. If yes, the entered cell's threat flag is set to 1 immediately on this first entry. If no, the standard three-entry conversion rule applies as in v0.11.2.

This is signature-matching with first-entry flagging on signature match. The signature is cell type alone: the architecture treats all HAZARD cells as members of the same category for the purposes of flag transfer. There is no perceptual feature, no adjacency pattern, no environmental marker — the type designation HAZARD is itself the signature.

The architectural claim being tested is that one experience of category-recognition (sufficient for the first three-entry conversion of any one hazard cell) is sufficient where three experiences of specific-cell-learning were previously required for each subsequent cell. The agent still has to encounter cells to flag them; the change is that flag conversion accelerates from three entries to one entry once the category has any flagged exemplar.

## 3. Specific implementation change

In the architecture's hazard-entry handling logic, the standard counter-and-threshold check is preceded by a category-membership check.

```python
def update_hazard_layer(self, cell):
    """v0.12: signature-matching flag conversion on cell-type signature."""
    # If the cell is not a hazard, nothing to do.
    if cell not in self.hazard_cells:
        return

    # If the cell is already flagged, nothing to do.
    if self.threat_flag.get(cell, 0) == 1:
        return

    # v0.12 addition: check whether any other hazard cell has flag = 1.
    # If yes, set this cell's flag to 1 immediately on this entry.
    any_other_flagged = any(
        self.threat_flag.get(h, 0) == 1
        for h in self.hazard_cells
        if h != cell
    )
    if any_other_flagged:
        self.threat_flag[cell] = 1
        return

    # v0.11.2 / v0.10 behaviour: standard three-entry conversion.
    self.hazard_entry_counter[cell] = self.hazard_entry_counter.get(cell, 0) + 1
    if self.hazard_entry_counter[cell] >= HAZARD_FLAG_THRESHOLD:
        self.threat_flag[cell] = 1
```

The single architectural change is the category-membership check. Before v0.12, each hazard cell required three entries to convert. Under v0.12, the first hazard cell still requires three entries (because no other hazard is flagged at the time of its first three encounters); every subsequent novel hazard cell the agent encounters is flagged on first entry, because by the time the agent reaches it, at least one hazard of the same category has already been flagged.

## 4. What v0.12 does not change

All other architectural elements from v0.11.2 are preserved unchanged:

- The threat layer's per-cell `threat_flag` dictionary, with frame cells pre-flagged at agent creation and hazard cells initialised to zero
- The hard-gate action-selection rule: any action targeting a flagged cell is excluded from the action-selection pipeline before Q-value computation, primitive bias computation, or epsilon-greedy random selection
- The hazard cost mechanism: each entry into an unflagged hazard cell delivers the configured hazard cost as a negative addition to the agent's intrinsic reward stream (note: under v0.12, the first hazard's three pre-flag entries pay cost as before; subsequent hazards' first entries pay cost once because the flag converts on that entry, after which the cell is not entered again)
- The mastery layer in full: per-attractor visit counter, mastery flag, MASTERY_THRESHOLD = 3, MASTERY_BONUS = 1.0
- The four mastery interventions from the v0.11 series: zero feature reward post-mastery, cleared attraction bias for mastered cells, preference reset at banking, blocked preference accumulation post-banking
- Three-phase developmental schedule, drive composition at each phase, learning parameters (rate 0.1, discount 0.9, epsilon 0.1)
- The pre-wired aversion bias for frame cells (-5.0), retained
- All metrics from v0.10 and v0.11.2

The single-variable-change discipline holds: v0.12 changes exactly one thing relative to v0.11.2 — the flag-conversion rule in the threat layer is augmented with a category-membership check.

## 5. Experimental matrix

The experimental matrix is one architecture (v0.12) crossed with six cost levels crossed with three run lengths crossed with ten runs per cell, totalling 180 runs.

The cost levels are 0.1, 0.5, 1.0, 2.0, 5.0, and 10.0, retained from v0.10 and v0.11.2 to enable direct comparison across the threat-layer programme.

The run lengths are 20,000, 80,000, and 160,000 steps, retained for the same reason and specifically to test whether the 20,000-step boundary effect is eliminated under v0.12.

For each (cost, run length, run index) triple, the random seed is matched to the v0.11.2 batch at the same condition. This produces matched-seed comparison: the v0.11.2 and v0.12 runs at the same condition share initial exploration trajectories until the threat layer first intervenes differently in v0.12 (which cannot be earlier than the first hazard cell's third entry, the same step at which the v0.11.2 run first flags). Comparison is therefore within-seed up to the point at which any second hazard cell is encountered post-first-flag, with the divergence after that point attributable to the signature-matching rule specifically.

The v0.11.2 architecture batch results (Baker, 2026g) provide the comparison baseline. No re-run of v0.11.2 is required.

## 6. Metrics

All v0.10 metrics and all v0.11.2 metrics are retained.

Specifically retained from v0.10:

- Hazard entries by phase (Phase 1, Phase 2, Phase 3, total)
- Phase 3 cleanness (fewer than three hazard entries during Phase 3)
- Full-run cleanness
- Number of hazard cells flagged per run
- Time to first flag, time to final flag
- Cost paid up to final flag
- Actions gated by threat layer per phase

Specifically retained from v0.11.2:

- Per-attractor visit counts at end of run
- Mastery sequence (the order in which attractors crossed the mastery threshold)
- First-banked attractor identity
- Mean attractors mastered per run
- Phase 3 visits to mastered vs unmastered attractors
- Distinct top-attractor count per condition (individuation signal 1)
- Most-common-mastery-sequence proportion (individuation signal 2)
- Rank-vs-distance Pearson correlation (individuation signal 3)

New to v0.12:

- **Time from first flag to second flag.** Under v0.11.2 and v0.10, this is bounded below by the time the agent takes to enter a second hazard cell three times after the first hazard's flag has converted. Under v0.12, this is bounded below by the time the agent takes to enter any second hazard cell once after the first hazard's flag has converted. The reduction in this metric is the direct mechanistic signature of the v0.12 architectural change.
- **First-entry flag conversions per run.** Under v0.11.2, this metric is zero by construction (every flag requires three entries). Under v0.12, this metric counts the number of hazard cells whose flag converted on first entry (i.e., not the first hazard cell to be flagged, but every subsequent flagged hazard cell). The pre-registered prediction is that this metric will equal (number of hazards flagged per run − 1) for every run in which any hazard is flagged.
- **Cost paid on signature-matched hazards.** Under v0.11.2, the cost paid per flagged hazard is bounded at three units of cost (three pre-flag entries). Under v0.12, the cost paid on signature-matched hazards is bounded at one unit of cost (the single first entry that triggers conversion). The reduction in cumulative cost relative to v0.11.2 is the cost-economics signature of the v0.12 architectural change.

Per-run numerical data is preserved in CSV format for post-hoc analysis. Aggregate findings are reported per-cost-level and per-run-length in the v0.12 paper.

## 7. Pre-registered interpretation categories

Six interpretation categories are pre-registered before the batch is run. Assignment is based on the metrics specified in Section 6, with the cost-5.0-at-20,000-steps outlier elimination as the primary measurement target and the v0.10 / v0.11.2 audits as secondary preservation tests.

**Category A — full elimination of outlier with full preservation.** v0.12 produces all three of the following:

- Phase 3 cleanness ≥ 9/10 at cost 5.0 at 20,000 steps (the outlier condition)
- Phase 3 cleanness ≥ 55/60 at 160,000 steps across all six cost levels (v0.10's Category A threshold)
- Mean attractors mastered ≥ 5.5/6 at 160,000 steps, AND v0.11.2's individuation Category F not triggered (≥4 distinct first-banked attractors, dominant mastery sequence < 40%, rank-vs-distance Pearson |r| < 0.7)

Category A is the pre-registered success condition. It requires that v0.12 eliminates the v0.10 outlier, preserves v0.10's full-stabilisation pattern, and preserves v0.11.2's mastery and individuation findings.

**Category B — partial elimination of outlier.** v0.12 produces Phase 3 cleanness of 7/10 or 8/10 at cost 5.0 at 20,000 steps, with v0.10 and v0.11.2 audits otherwise preserved at Category A thresholds.

**Category C — no meaningful improvement on outlier.** v0.12 produces Phase 3 cleanness of 6/10 or fewer at cost 5.0 at 20,000 steps. The signature-matching rule did not produce the predicted resolution mechanism.

**Category D — over-flagging.** Any of the following:

- Non-hazard cells flagged at any point during any run
- All hazards in any run flagged within the first 1,000 steps of Phase 2
- Mean first-entry flag conversions per run substantially exceeds the predicted (number flagged − 1) — i.e., flags appearing on cells the agent has not entered

The first sub-condition tests that the category-membership check correctly restricts flag transfer to hazard cells. The second sub-condition tests that the rule does not produce categorical avoidance within the prescribed-traversal phase, which would indicate that signature-matching is operating before the agent has had the opportunity to learn any hazard at all (which the architecture cannot do, given that the first hazard still requires three entries; if observed, this would indicate an implementation bug). The third sub-condition tests that flags appear only on cells the agent has actually entered.

**Category E — developmental degradation.** Any v0.10 secondary signature (concentration, attractor ratio) or any v0.11.2 secondary signature (mean attractors mastered, mastery dynamics, individuation) is substantially reduced under v0.12 compared with the relevant baseline. "Substantially reduced" is defined as: mean attractors mastered drops below 5.0/6 at 160,000 steps, OR distinct first-banked attractors drops to ≤3 of 6 at 160,000 steps, OR most-common-mastery-sequence proportion exceeds 30% at 160,000 steps, OR Phase 3 attractor visit ratio drops below 5× chance at 160,000 steps.

**Category F — interpretation collapse: success attributable to existing type representation.** v0.12 produces Category A or Category B at the outlier-elimination level (i.e., the cost-5.0 outlier is eliminated or substantially reduced), but the mechanistic interpretation is bounded by the cell-type-only signature choice. The architecture already represents cell types directly through the `hazard_cells` set; the v0.12 architectural change is the rule that allows this existing information to act on flag conversion. Whether the success demonstrates "category-level generalisation works in this architecture" or "the architecture's existing type representation was sufficient once the rule allowed it to act" cannot be distinguished from the v0.12 batch alone.

Category F is not a failure condition in the operational sense — the outlier may be eliminated and the audits may be preserved. Category F is a pre-registered honesty constraint on the architectural claim. If Category F applies, the v0.12 paper will report the operational success while explicitly bounding the architectural claim, and will name the v0.14+ iteration that tests richer signature options (adjacency patterns, environmental markers) as the necessary follow-on for distinguishing the two readings.

Category F is determined by the cell-type-only signature choice itself, not by the data. The Category F honesty constraint applies to any v0.12 result in which the outlier is eliminated. The pre-registration commits in advance to handling the architectural claim accordingly.

## 8. Predicted result and the architectural claim being made

The pre-registered prediction is Category A.

The mechanistic prediction is that under v0.12, agents will pay cost on the first hazard cell encountered (three pre-flag entries, as in v0.11.2), then flag every subsequent novel hazard cell on first entry. The cost-5.0-at-20,000-steps boundary effect arises in v0.10 / v0.11.2 because late-encountered hazards need two further entries after their first to convert; under v0.12 these later entries do not occur because the cell is flagged on first entry and the hard gate excludes it from action selection on subsequent steps. The time-budget pressure that produces the boundary effect at short run lengths is therefore eliminated.

The cost-economics prediction is that v0.12 will pay substantially less cumulative cost than v0.11.2 across the matrix, with the magnitude of the reduction depending on how many hazards are encountered per run. At long run lengths, where v0.11.2 already pays bounded cost per flagged hazard, the v0.12 reduction will be smaller in absolute terms. At short run lengths, where v0.11.2 may not complete flag conversion on all encountered hazards, the v0.12 reduction will be larger.

The behavioural-preservation prediction is that v0.11.2's mastery findings will be unaffected by the v0.12 change. The mastery layer is not modified. The threat layer's interaction with the mastery layer is structural (flagged cells are excluded from action selection, including for any attractor adjacency considerations), and the v0.12 change accelerates flagging without changing what flagging does. There is no architectural pathway by which v0.12 should disturb mastery dynamics. If v0.12 produces Category E, this would be an unexpected finding requiring its own characterisation.

The architectural claim being made — and the claim Category F is designed to bound — is that signature-matching with first-entry flagging on cell-type signature constitutes category-level generalisation in the threat layer. This claim is at risk of being weaker than it appears, because the architecture's existing representation of cell types is sufficient information for flag transfer once the rule permits it. The pre-registration commits in advance to reporting this honestly. The richer-signature work reserved for v0.14+ (adjacency patterns, environmental markers, perceptual features added to the environment) is the necessary follow-on to distinguish "categorical generalisation works in this architecture" from "the cell-type representation was sufficient once unlocked."

## 9. Methodological note

This is a pre-registration for a new iteration, not an amendment to a previous pre-registration. The v0.11 series produced three documents (pre-registration, v0.11.1 amendment, v0.11.2 amendment) because each implementation revealed an aspect of the architectural goal that had not been anticipated at design time. The methodological observation that emerged from the v0.11 series — that single architectural goals can decompose into multiple code-path interventions, and that single-pass implementation can miss perspective the multi-pass surfaces — is part of the methodological record and informs the design of v0.12.

Specifically, the v0.12 design attempts to anticipate the implementation-time discoveries that v0.11 surfaced only through iteration. The four open considerations from the v0.12 design conversation (Baker, 2026, internal record, 25 April) were settled before this pre-registration was written:

- The kind of generalisation v0.12 tests is signature-matching with first-entry flagging on signature match, not one-shot full generalisation and not graded probabilistic prior. The choice was made on the grounds that signature-matching directly addresses the v0.10 outlier mechanism, preserves the experience-driven character of the architecture, and maintains single-variable-change discipline.
- The signature is cell type alone, not adjacency pattern and not environmental marker. The choice was made on the grounds that cell-type-only is the simplest test of whether category-level generalisation works at all, and that richer signatures are appropriate for v0.14+ once the simpler version is established.
- The successful-avoidance bonus is deferred. Adding it would constitute a second architectural change, breaking single-variable-change discipline. It is reserved for a subsequent iteration.
- The Category F honesty constraint is named explicitly in this pre-registration to handle the cell-type-only signature concern in advance, rather than retrofitting the architectural claim at interpretation time.

The decomposition of the design conversation into pre-registered choices is the methodological response to v0.11's lesson. Whether this anticipation succeeds — whether v0.12 implements cleanly in a single pass without surfacing further architectural questions — is itself an empirical question this iteration tests.

## 10. Stopping rule

If the v0.12 implementation produces an implementation-time discovery requiring further architectural change, a v0.12.1 amendment may be issued, with all v0.11.2 amendment-discipline conditions applying: amendment committed before amended code is written, single architectural change per amendment, full retention of original pre-registration in the public repository.

The amendment budget for v0.12 is fresh: three amendments within the iteration is the limit, beyond which the iteration's design itself is what needs revision. The v0.11 series exhausted its amendment budget at v0.11.2 because each amendment surfaced a further code-path intervention required by the same architectural goal. The v0.12 design has been examined with that lesson in mind, and the four open design considerations have been settled before pre-registration. The fresh amendment budget is reasonable rather than an erosion of discipline because the design discipline has been tightened in advance.

If a v0.12.1 amendment is required, the question becomes whether signature-matching is the right reading of the v0.12 architectural goal at all, not whether incremental amendments can resolve specific implementation issues. The stopping rule is named here to preserve this discipline against the temptation to treat amendments as a permitted means of saving the iteration's design.

## 11. Commitments

Three methodological commitments accompany this pre-registration:

First, no v0.12 code will be written before this pre-registration document is committed to the public repository at github.com/RancidShack/developmental-agent.

Second, the pre-registration specifies a single architectural change: the flag-conversion rule in the threat layer is augmented with a category-membership check on cell-type signature. Any implementation-time discovery requiring additional architectural change will trigger the amendment procedure with the conditions and stopping rule named in Section 10.

Third, the v0.12 batch results will be reported alongside both the v0.10 batch (for threat-layer comparison) and the v0.11.2 batch (for mastery-and-individuation comparison). The full inheritance arc from v0.10 → v0.11.2 → v0.12 is part of the research contribution: persistent representation (v0.10) plus bounded mastery (v0.11.2) plus category-level generalisation (v0.12) as a programmatic answer to the unbounded-accumulation problem v0.9 identified.
