# v0.9 Pre-Registration: Cost-Based Hazards

**Project:** Developmental Agent
**Version:** v0.9
**Repository:** github.com/RancidShack/developmental-agent
**Document committed:** [date of commit, prior to first v0.9 run]
**Builds on:** v0.8 (batched, documented, ten-run individuation result)

---

## 1. Background

v0.8 demonstrated that the developmental agent produces zero rule violations, focused autonomy with concentration above 50x uniform across all runs, feature alignment in every run, and full biographical individuation with all six attractors emerging as primary focus across the ten-run batch. Rules in v0.8 were enforced architecturally — hazard cells were impassable by construction. The agent's compliance with those rules was therefore not directly testable as a learned behaviour; compliance and enforcement were confounded.

v0.9 separates them.

## 2. Research Question

Does the rule-compliance observed in v0.8 reflect *internalised* avoidance, or was it merely *enforced* by the architecture of the environment?

## 3. Hypothesis

If avoidance was internalised, learned avoidance behaviours will persist when the architectural floor is removed and replaced with a cost gradient. If avoidance was purely architectural, behaviours will collapse immediately under any cost-based regime. If internalisation is partial, collapse will occur at identifiable cost thresholds, and those thresholds may vary across biographically individuated agents.

## 4. Pre-Specified Outcome Categories

Three outcomes are defined in advance. Each is a meaningful result.

**4.1 Full survival.** Avoidance behaviours persist across all six cost levels, including the lowest (0.1). The agent treats the rule as binding regardless of cost. Interpretation: avoidance was genuinely internalised; the architectural enforcement in v0.8 was redundant rather than constitutive.

**4.2 Full collapse.** Avoidance behaviours fail at the lowest cost level (0.1) and remain failed across all higher levels. The agent treats the rule as non-binding once the architectural floor is removed. Interpretation: v0.8's compliance was an artefact of enforcement; nothing was internalised.

**4.3 Threshold-located partial collapse.** Avoidance behaviours persist at low cost levels and fail at higher cost levels, with a detectable transition. Interpretation: internalisation is real but bounded; the threshold value locates the boundary at which preference overrides learned avoidance. This is the most informative outcome.

A fourth, secondary question runs across all three categories: do biographically distinct agents — those whose v0.8-equivalent runs produced different primary attractors — show different threshold values? Evidence of attractor-specific thresholds would extend v0.8's individuation finding into the domain of rule-internalisation strength.

## 5. Experimental Design

**5.1 Cost function.** Scaled by magnitude across structurally similar hazards. Costs felt only on contact — no proximity signal, no anticipatory information. Six cost levels, log-spaced: 0.1, 0.5, 1.0, 2.0, 5.0, 10.0.

**5.2 Control condition.** Ten runs with v0.8-style impassable hazards, executed in the same batch as the cost-based runs, using the same codebase with the `hazard_mode` flag set to `impassable`. This permits direct same-batch comparison rather than reliance on the v0.8 batch as historical baseline.

**5.3 Run counts.** Ten runs per cost level across six levels (sixty cost-based runs) plus ten control runs. Total: seventy runs.

**5.4 Run length.** Matched exactly to v0.8. No other parameter changes.

**5.5 Architecture.** Single codebase extended from v0.8. The only modification is the hazard handler, governed by a `hazard_mode` config flag with values `impassable` (v0.8 behaviour) and `cost` (v0.9 behaviour). All other code paths unchanged.

## 6. Threshold-Crossing Definition

A run is classified as "rule violated" if the agent enters cost-bearing cells three or more times during the run. A cost level is classified as "threshold crossed" if a majority of runs at that level meet the rule-violated criterion.

This definition is fixed before any v0.9 data exists. It will not be adjusted post hoc.

## 7. Refinement Conditions

The six initial cost levels are a first-cut grid. A refinement batch will be run if, and only if, one of the following conditions is met:

- The threshold lies at or above 5.0. A second batch will extend the upper range (10, 20, 50) to locate where collapse actually occurs.
- The threshold lies at or below 0.5. A second batch will densify the spacing below 1.0 to resolve the lower-end threshold.
- The threshold falls between 1.0 and 2.0. No refinement batch is required; the initial grid resolved the question.

Refinement batches will be run as separate, additionally pre-registered increments. The initial v0.9 batch is not contingent on refinement results.

## 8. What Would Count as a Null Result

A null result is defined as: agents at all cost levels showing behaviour statistically indistinguishable from each other and from the impassable control on the threshold-crossing measure. This would indicate that the hazard manipulation was insufficient to produce any behavioural variation, suggesting either that the cost range was inadequate or that the agent's reward structure does not interact with the cost mechanism in the expected way. A null result would prompt re-examination of the cost-function design rather than an immediate refinement batch.

## 9. What Is *Not* Being Tested in v0.9

To preserve the cleanness of the threshold finding, the following are deliberately excluded from v0.9 and reserved for later phases:

- Proximity-ramped costs (anticipatory avoidance). Reserved for v1.0.
- Developmental progression / readiness-gated environment expansion. Separate research arc.
- Non-tabular representation / function approximation. Separate research arc.
- Multi-agent dynamics. Separate research arc.

## 10. Reporting Commitment

All seventy runs will be reported regardless of outcome. Per-run threshold classifications will be released with the eventual preprint. The dataset will be committed to the repository alongside the analysis code. Negative and null results will be reported with the same prominence as positive results.
