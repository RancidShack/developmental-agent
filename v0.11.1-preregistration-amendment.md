# v0.11.1 Pre-Registration Amendment: Preference Reset on Mastery

**Author:** Nicholas P M Baker, Synapstak Ltd
**Date:** 24 April 2026
**Status:** Amendment to v0.11 pre-registration, committed before any v0.11.1 code is written
**Preceding work:** v0.11 pre-registration (Baker, 2026e), v0.10 (Baker, 2026d), v0.9 (Baker, 2026c), v0.8 (Baker, 2026a)

---

## 1. Reason for amendment

The v0.11 architecture was implemented per the pre-registered specification and tested via single-run probe at HAZARD_COST=1.0, MASTERY_THRESHOLD=3, MASTERY_BONUS=1.0, NUM_STEPS=20000. The probe revealed an architectural interaction not anticipated by the pre-registration.

Per-attractor visit counts at end of run:

```
(3, 3):    69 [MASTERED]
(4, 15):   79 [MASTERED]
(9, 10):   33 [MASTERED]
(11, 5):   29 [MASTERED]
(15, 16):   1
(16, 3): 1000 [MASTERED]
```

The attractor at (16, 3) was visited 1,000 times despite being mastered (banked) early in the run. Phase 3 visit distribution was 69 visits to mastered attractors and 0 visits to unmastered ones — the inverse of the central hypothesis prediction. The remaining unmastered attractor at (15, 16) was visited only once across the entire 20,000-step run.

The mechanism producing this behaviour is identifiable. The v0.11 architecture cleared the primitive ATTRACTION_BONUS for mastered attractors (per Section 4.5 of the v0.11 pre-registration) and set feature_reward to zero post-mastery (per Section 4.2). However, the cell_preference accumulator was not affected by mastery. Pre-mastery visits accumulated preference from the per-visit feature reward (0.15 × N=3 visits = 0.45) plus the one-time MASTERY_BONUS at banking (1.0), depositing approximately 1.45 units of preference mass on the cell at the moment of banking. This preference mass persisted after mastery and continued to accumulate from learning-progress reward on subsequent visits.

When Phase 3 began (preference_weight = 0.8), the accumulated preference on mastered attractors produced a strong action-selection signal that exceeded the cleared attraction bias and the zeroed feature reward. The agent was pulled back to the cell with the highest accumulated preference and remained there. The mastery mechanism succeeded in driving rapid mastery of multiple attractors, but the developmental trajectory did not progress to the unmastered remainder.

This is a single-probe finding, but the mechanism is architecturally evident rather than stochastic. Running the full batch under the original v0.11 specification would produce variation in *which* mastered attractor each agent fixates on, but the qualitative pattern — Phase 3 fixation on one mastered attractor rather than continued exploration toward unmastered ones — would be expected to recur across runs.

## 2. The amendment

The v0.11.1 architecture introduces one additional architectural change to the v0.11 specification: **on mastery, the cell's accumulated preference is reset to zero**.

This makes the mastery mechanism fully architecturally coherent across three signal types:

- Feature reward: drops to zero post-mastery (already specified in v0.11)
- Primitive attraction bias: clears for mastered cells (already specified in v0.11)
- Cell preference: resets to zero at the moment of banking (new in v0.11.1)

After mastery, a banked attractor produces no signal pulling the agent back. The cell remains visitable, but no architectural mechanism reinforces returning to it.

## 3. Specific implementation change

In the `update_mastery_layer` method, when an attractor reaches MASTERY_THRESHOLD visits and the mastery_flag is set:

```python
if (self.attractor_visit_counter[entered_cell] >= MASTERY_THRESHOLD
        and self.mastery_flag.get(entered_cell, 0) == 0):
    self.mastery_flag[entered_cell] = 1
    self.cell_preference[entered_cell] = 0.0   # v0.11.1 addition
    self.mastery_order_sequence.append(entered_cell)
    if self.time_to_first_mastery is None:
        self.time_to_first_mastery = step
    self.time_to_final_mastery = step
```

The reset applies only at the banking step. Subsequent visits to the mastered cell still accumulate preference from learning-progress reward (cell_preference is incremented in update_model based on r_progress + r_feature, and post-mastery r_feature is zero, so the only preference accumulation post-mastery is from learning progress, which should be near zero for a well-modelled cell).

Note: the MASTERY_BONUS is still added to the intrinsic reward stream at the banking step (and so flows into Q-value updates), but the bonus does not enter cell_preference directly — preference accumulates only the standard r_progress + r_feature, and at the banking step r_feature returns the bonus. So the banking step deposits the bonus into preference, then the reset zeroes it. The net effect on preference at the banking step is zero. The bonus remains visible in Q-values via the intrinsic reward update.

This is a deliberate design choice. The bonus is rewarded behaviourally (via Q-values, which influence action selection going forward) but is not stored as long-term preference (which would create the problem v0.11 demonstrated).

## 4. What v0.11.1 does not change

All other v0.11 architectural elements are preserved unchanged:

- Three-phase developmental schedule
- Drive composition at each phase (novelty, progress, preference, feature weights)
- Learning parameters (rate 0.1, discount 0.9, epsilon 0.1)
- The threat layer and hazard handling (inherited from v0.10)
- Phase 1 prescribed traversal
- The attractor visit counter and mastery_flag mechanism
- The MASTERY_THRESHOLD = 3 and MASTERY_BONUS = 1.0 parameters
- The clearing of primitive ATTRACTION_BONUS for mastered cells
- The depletion of feature_reward on mastered cells
- All metrics from v0.10 and v0.11

The single-variable-change discipline holds: v0.11.1 changes exactly one thing relative to v0.11 — the reset of cell_preference at the mastery step.

## 5. Predicted behavioural change

Under v0.11.1, the predicted developmental trajectory matches the original v0.11 central hypothesis: agents systematically work through attractors, mastering each via N=3 visits, and after mastery the cell loses all signal that would draw the agent back. Phase 3 visits should distribute toward unmastered attractors when any remain, and toward post-mastery exploration of neutral cells once all attractors are banked.

The pre-registered Categories A through F from the v0.11 pre-registration apply unchanged to v0.11.1's measurement. Specifically:

- Category A (full mastery trajectory) becomes the primary expectation
- Category B (partial mastery) remains a meaningful intermediate outcome
- Category E (developmental degradation) is now less likely, since the mechanism that produced the v0.11 fixation is removed
- Category F (individuation collapse) remains an open question that the batch will measure directly

## 6. Methodological note

This amendment follows the discipline established for v0.9: when implementation-time discovery reveals an architectural interaction that the pre-registration did not anticipate, the response is to commit an explicit amendment document before any amended computation is run, with full reasoning for what changed and why.

The original v0.11 pre-registration is retained in the repository unchanged. The v0.11 single-run probe code (curiosity_agent_v0_11.py) is retained as the implementation of the original specification. v0.11.1's implementation will be a new file (curiosity_agent_v0_11_1.py) with the amendment applied. The batch will run against v0.11.1, with v0.10 as the matched-seed comparison, matching the experimental matrix specified in the v0.11 pre-registration.

The probe finding under v0.11 — that bounded categorical mastery alone does not produce the predicted developmental trajectory because pre-mastery preference accumulation persists after banking — is itself a substantive finding worth recording. It will be reported in the v0.11 paper alongside the v0.11.1 batch results, framed as the architectural discovery that motivated the amendment.

## 7. Commitments

Three methodological commitments accompany this amendment:

First, no v0.11.1 code will be written before this amendment document is committed to the public repository.

Second, the amendment specifies a single architectural change. Any further implementation-time discovery requiring additional architectural change will be handled through a further amendment, not through silent revision.

Third, the v0.11.1 batch results will be reported alongside the v0.11 probe finding, so the research record shows both the original design's behaviour and the amended design's behaviour. The progression from v0.11 to v0.11.1 is itself part of the research contribution.
