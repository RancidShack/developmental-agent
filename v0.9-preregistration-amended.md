# v0.9 Pre-Registration (Amended): Cost-Based Hazards and Aversion Internalisation

**Project:** Developmental Agent
**Version:** v0.9
**Repository:** github.com/RancidShack/developmental-agent
**Document committed:** 22 April 2026, prior to first v0.9 run
**Amends:** v0.9-preregistration.md (committed 22 April 2026, hash 666b563)
**Builds on:** v0.8 (batched, documented, ten-run individuation result)

## 1. Reason for Amendment

The original v0.9 pre-registration (committed earlier today) framed the experiment as a test of whether v0.8's observed rule-compliance was *internalised* or *enforced*, with a single batch manipulating the hazard handler.

On subsequent review of the v0.8 code prior to writing any v0.9 implementation, it became clear that hazard avoidance in v0.8 is produced by three distinct mechanisms, not one:

1. **Architectural impassability.** The world refuses to move the agent into hazard cells (`world.step()` returns `success=False`).
2. **Pre-wired aversion bias.** HAZARD cells trigger a large negative Q-value bias (`AVERSION_PENALTY = -5.0`) at action selection, operating on perception of adjacent cell types rather than experience.
3. **Epsilon-exploration filter.** Random exploration excludes hazard-directed actions before any randomness is applied.

The v0.8 agent therefore never experiences the consequence of approaching a hazard; avoidance is not learned through cost, it is pre-installed and architecturally enforced. The original pre-registration's framing — "was the rule internalised through learning?" — implicitly assumed a learning process that v0.8 does not contain.

A single-batch study weakening only one of the three mechanisms cannot adequately separate their contributions. The study has accordingly been restructured into two batches testing two related but distinct questions, pre-registered before any v0.9 code runs.

The original pre-registration document is retained in the repository unchanged. This amended document supersedes it as the operative experimental design.

## 2. Revised Research Questions

**Question A.** Does the pre-wired aversion bias alone sustain hazard avoidance when architectural impassability is removed?

**Question C.** Can hazard avoidance be learned from experienced cost alone, without a pre-wired aversion bias?

The two questions isolate the contributions of scaffolding and wall to the v0.8 result.

## 3. Design A: Removal of Architectural Impassability

**Manipulation.** The world permits entry into hazard cells, applying a scalar cost to the intrinsic reward signal on entry. Pre-wired aversion bias retained (HAZARD still triggers `AVERSION_PENALTY = -5.0`). Epsilon-exploration filter retained.

**What this tests.** Whether the pre-wired bias is sufficient to sustain avoidance in the absence of architectural enforcement, or whether the bias's effect in v0.8 depended on never being challenged by actual experience of hazard entry.

**FRAME cells are unchanged.** Frames remain architecturally impassable and retain their aversion bias. Frames represent the edge of the world — an ontological boundary, not a normative rule. Testing whether an agent can learn to stay inside a bounded environment is not the question of interest.

**Cost levels.** Six log-spaced values: 0.1, 0.5, 1.0, 2.0, 5.0, 10.0. Costs felt only on contact; no proximity signal or anticipatory information.

**Runs.** Ten runs per cost level (sixty experimental runs) plus ten control runs with full v0.8 hazard mechanism (`hazard_mode = impassable`). Total for Design A: seventy runs.

## 4. Design C: Removal of Pre-Wired Aversion

**Manipulation.** World permits entry into hazard cells at cost (as in Design A). HAZARD-specific aversion bias removed (`AVERSION_PENALTY` for HAZARD set to 0.0; FRAME aversion retained unchanged). Epsilon-exploration filter modified to exclude frame-directed actions only, not hazard-directed actions.

**What this tests.** Whether the agent can develop avoidance behaviour purely from experienced cost, without the pre-wired scaffolding. This is the closer approximation to genuine learning-based internalisation.

**FRAME cells remain fully protected.** Frame aversion retained to preserve the ontological boundary. Only the normative rule (HAZARD) is exposed to the learning-from-experience test.

**Cost levels.** Six log-spaced values: 0.1, 0.5, 1.0, 2.0, 5.0, 10.0. Identical to Design A for direct comparison.

**Runs.** Ten runs per cost level (sixty experimental runs) plus ten control runs with impassable hazards *and* no HAZARD aversion (`hazard_mode = impassable_no_aversion`). This second control is important: it tests whether removing HAZARD aversion changes behaviour even when hazards remain impassable, establishing a baseline for C independent of the cost manipulation. Total for Design C: seventy runs.

## 5. Total Batch Size

Seventy runs for Design A plus seventy runs for Design C equals 140 runs total. Computation assessed as feasible on user's hardware (MacBook M4).

## 6. Pre-Specified Outcome Categories

The three outcome categories from the original pre-registration apply to both designs independently:

**Full survival.** Avoidance behaviours persist across all six cost levels. Interpretation for A: pre-wired bias is sufficient. Interpretation for C: cost-based learning is sufficient to produce avoidance.

**Full collapse.** Avoidance fails at the lowest cost level. Interpretation for A: bias without architectural enforcement is inadequate. Interpretation for C: cost-based learning does not produce avoidance within run length.

**Threshold-located partial collapse.** Avoidance persists at low costs, fails at higher costs, with a detectable transition. The threshold value locates the boundary at which preference overrides the remaining avoidance mechanism.

A secondary question applies across both designs: do biographically distinct agents — those whose v0.8-equivalent runs produced different primary attractors — show different threshold values? Evidence of attractor-specific thresholds in either design would extend v0.8's individuation finding into the domain of rule-internalisation strength.

A *cross-design* question: how do A and C compare at equivalent cost levels? If A survives cleanly and C collapses, pre-wired scaffolding is load-bearing. If A collapses and C survives (unlikely but possible), experienced cost is a stronger teacher than pre-wired bias. If both collapse at similar thresholds, the two mechanisms produce functionally equivalent avoidance strength. If both survive cleanly across all cost levels, v0.8's result is robust under substantially weakened constraint of either kind.

## 7. Threshold-Crossing Definition

Unchanged from original pre-registration. A run is classified as "rule violated" if the agent enters cost-bearing cells three or more times during the run. A cost level is classified as "threshold crossed" if a majority of runs at that level meet the rule-violated criterion. This definition is fixed before any v0.9 data exists and will not be adjusted post hoc.

## 8. Refinement Conditions

Unchanged from original pre-registration. Refinement batches contingent on where the threshold lands:

- Threshold at or above 5.0: extend upper range (10, 20, 50) in a follow-up batch.
- Threshold at or below 0.5: densify spacing below 1.0 in a follow-up batch.
- Threshold between 1.0 and 2.0: no refinement batch required.

Refinement may be triggered independently for Design A and Design C.

## 9. Implementation Sequence

Design A implemented and run first. Reasons:

- A is a smaller deviation from v0.8 (single change to the hazard handler). Verification that the `hazard_mode` config refactor preserves v0.8 behaviour on `hazard_mode = impassable` is cleaner when fewer other variables have changed.
- Once A is producing sensible data, C is an additional small change (nulling HAZARD aversion, lifting HAZARD filter) on top of an already-verified codebase. This sequencing isolates sources of unexpected behaviour should any arise.

Design C implementation begins only after Design A completes and initial sanity checks pass.

## 10. Config Structure

A single `hazard_mode` flag in the agent's configuration, with four values:

- `impassable` — v0.8 behaviour: wall, aversion, filter. Control for Design A.
- `impassable_no_aversion` — wall, no HAZARD aversion, no HAZARD filter. Control for Design C.
- `cost` — no wall, aversion, filter. Experimental condition for Design A.
- `cost_no_aversion` — no wall, no HAZARD aversion, no HAZARD filter. Experimental condition for Design C.

FRAME handling is unchanged across all four modes.

## 11. What Is *Not* Being Tested in v0.9

Unchanged from original. Deliberately excluded from v0.9 and reserved for later phases:

- Proximity-ramped costs (anticipatory avoidance). Reserved for v1.0.
- Developmental progression / readiness-gated environment expansion. Separate research arc.
- Non-tabular representation / function approximation. Separate research arc.
- Multi-agent dynamics. Separate research arc.
- Proto-selfhood / attunement-rupture architectures. Parked for potential v2.0 research arc.

## 12. Null Result Definition

Unchanged from original. A null result is: agents at all cost levels showing behaviour statistically indistinguishable from each other and from the relevant control on the threshold-crossing measure. For Design A this would indicate the cost range failed to bite against the residual bias. For Design C it would indicate the agent is neither learning to avoid nor being pushed to violate — which would prompt re-examination of whether the v0.8 reward structure provides any signal under which hazard entry could matter to the agent.

## 13. Reporting Commitment

All 140 runs will be reported regardless of outcome. Per-run threshold classifications will be released with the eventual preprint. The dataset will be committed to the repository alongside the analysis code. Negative and null results will be reported with the same prominence as positive results. Both Design A and Design C results will be reported in the same document, regardless of which (if either) produces a clean threshold finding.
