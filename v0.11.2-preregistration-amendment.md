# v0.11.2 Pre-Registration Amendment: Preference Accumulation Blocked on Mastered Cells

**Author:** Nicholas P M Baker, Synapstak Ltd
**Date:** 24 April 2026
**Status:** Second amendment to v0.11 pre-registration, committed before any v0.11.2 code is written
**Preceding work:** v0.11.1 amendment (Baker, 2026f), v0.11 pre-registration (Baker, 2026e), v0.10 (Baker, 2026d)

---

## 1. Reason for amendment

The v0.11.1 architecture was implemented per the v0.11.1 amendment specification and tested via single-run probe at HAZARD_COST=1.0, MASTERY_THRESHOLD=3, MASTERY_BONUS=1.0, NUM_STEPS=20000. The probe revealed that the v0.11.1 amendment, while correctly identifying the architectural problem (preference persistence on mastered cells), addressed it incompletely.

Per-attractor visit counts at end of run under v0.11.1:

```
(3, 3):    1
(4, 15):  30 [MASTERED]
(9, 10):   5 [MASTERED]
(11, 5):  25 [MASTERED]
(15, 16): 944 [MASTERED]
(16, 3):  67 [MASTERED]
```

Phase 2 mastery dynamics improved substantially under v0.11.1: five attractors mastered by step 3,506, with mastery order showing systematic spread across the environment. This confirmed that the depletion + bias clearing + banking-time preference reset combination produces the predicted developmental trajectory during Phase 2.

However, Phase 3 behaviour reproduced the v0.11 fixation pattern with a different cell. The agent visited (15, 16) 944 times after mastery, accumulating preference of 19.38 by end of run. Phase 3 visits to mastered: 27. To unmastered: 0. The remaining unmastered attractor at (3, 3) was not pursued.

The mechanism producing this is identifiable. The v0.11.1 amendment reset cell_preference to zero at the banking step. But on every subsequent visit, the standard preference accumulation in update_model continues to run: `self.cell_preference[cell] += r_progress + r_feature`. For mastered cells, r_feature is zero (per v0.11's depletion mechanism), but r_progress continues to accumulate small per-visit amounts. Across 944 visits, even small per-visit accumulations compound substantially. The accumulated preference on a mastered cell, multiplied by preference_weight = 0.8 in Phase 3, produces sufficient action-selection signal to pull the agent back, generating more visits, generating more preference, and reconstructing the fixation feedback loop.

The v0.11.1 amendment correctly identified that preference persistence was the architectural problem and correctly intended that mastered cells should produce no signal pulling the agent back. The implementation addressed the banking-time spike of preference but did not address ongoing post-mastery accumulation. This was an incomplete realisation of the amendment's stated intent.

## 2. The amendment

The v0.11.2 architecture introduces one further architectural change: **on mastered cells, preference accumulation is blocked entirely**. The cell's preference value remains at zero (the value set at the banking step under v0.11.1) for the remainder of the run.

Combined with v0.11 and v0.11.1, this means a banked attractor produces no signal pulling the agent back via any of the four mechanisms that influence action selection:

- Feature reward: zero post-mastery (v0.11)
- Primitive attraction bias: cleared for mastered cells (v0.11)
- Cell preference at banking: reset to zero (v0.11.1)
- Cell preference accumulation post-banking: blocked entirely (v0.11.2)

The mastered cell becomes truly inert with respect to all action-selection signals from the preference and feature systems. The cell remains visitable — the agent can pass through it during exploration — but no architectural mechanism reinforces returning to it.

## 3. Specific implementation change

In the `update_model` method, the preference accumulation line is made conditional on the cell not being mastered:

```python
def update_model(self, state, action, next_state, error, r_progress, r_feature):
    self.visit_counts[next_state] += 1
    self.forward_model[(state, action)][next_state] += 1
    self.fast_errors[(state, action)].append(error)
    self.slow_errors[(state, action)].append(error)
    cell = (next_state[0], next_state[1])
    if cell in self.scope:
        self.covered.add(cell)
    # v0.11.2: block preference accumulation on mastered cells.
    # Mastered cells should produce no signal pulling the agent back;
    # this means preference cannot accumulate via learning-progress
    # reward on post-banking visits.
    if self.mastery_flag.get(cell, 0) == 0:
        self.cell_preference[cell] += r_progress + r_feature
    # v0.11.1: if this cell was just mastered (banking step), reset
    # its preference to zero. Now redundant given v0.11.2's blocking,
    # but retained for clarity of architectural intent and as a
    # safeguard against any future code path that might bypass the
    # accumulation block.
    if (self.mastery_flag.get(cell, 0) == 1
            and self.attractor_visit_counter.get(cell, 0) == MASTERY_THRESHOLD):
        self.cell_preference[cell] = 0.0
```

The conditional accumulation is the substantive change. The v0.11.1 reset is retained for architectural clarity — it makes explicit that at the banking step, preference is zeroed — and as a safeguard for any future modification that might inadvertently bypass the accumulation block.

## 4. What v0.11.2 does not change

All other v0.11 and v0.11.1 architectural elements are preserved unchanged:

- Three-phase developmental schedule
- Drive composition at each phase
- Learning parameters (rate 0.1, discount 0.9, epsilon 0.1)
- Threat layer and hazard handling (inherited from v0.10)
- Phase 1 prescribed traversal
- Attractor visit counter and mastery_flag mechanism
- MASTERY_THRESHOLD = 3 and MASTERY_BONUS = 1.0 parameters
- Clearing of primitive ATTRACTION_BONUS for mastered cells
- Depletion of feature_reward on mastered cells
- All metrics from v0.10 and v0.11

The single-variable-change discipline holds: v0.11.2 changes exactly one thing relative to v0.11.1 — preference accumulation is blocked on mastered cells. The reset at banking, the bias clearing, the feature reward depletion, and the mastery counter all remain as previously specified.

## 5. Predicted behavioural change

Under v0.11.2, the predicted developmental trajectory is the original v0.11 central hypothesis: agents systematically work through attractors, mastering each via N=3 visits, and after mastery the cell loses all signal that would draw the agent back. Phase 3 should distribute visits toward unmastered attractors when any remain, and toward post-mastery exploration of neutral cells once all are banked.

The pre-registered Categories A through F from the v0.11 pre-registration apply unchanged. Specifically:

- Category A (full mastery trajectory) is the primary expectation
- Category B (partial mastery) remains a meaningful intermediate outcome
- Category C (weak mastery) is now considered unlikely given the Phase 2 mastery dynamics observed under v0.11.1
- Category E (developmental degradation) remains possible if blocking preference accumulation on mastered cells produces unintended interference with other developmental signatures
- Category F (individuation collapse) remains the open question that the batch will measure directly

## 6. Methodological note

This is the second amendment to the v0.11 pre-registration. The amendment sequence within v0.11 (v0.11 → v0.11.1 → v0.11.2) is unusual relative to v0.10's clean single-pass implementation. The unusualness is itself part of the methodological record.

Two observations about the sequence are worth naming:

First, each amendment was triggered by an explicit implementation-time discovery — the v0.11 probe revealed preference persistence; the v0.11.1 probe revealed that the banking-time reset was insufficient against ongoing accumulation. Each amendment was committed before the amended code was written. The original pre-registration and both amendments are retained in the public repository unchanged. The discipline of pre-registration before implementation has held throughout the sequence; the iterations have been honest responses to what each implementation revealed.

Second, the amendment sequence reveals a property of the v0.11 architectural goal: bounding the preference signal for mastered attractors requires intervention at multiple points in the code path, not at one. The v0.11 pre-registration described this as "no architectural mechanism reinforces returning" — which turned out to require:

(a) zero feature reward post-mastery (v0.11)
(b) cleared attraction bias (v0.11)
(c) preference reset at banking (v0.11.1)
(d) blocked preference accumulation post-banking (v0.11.2)

Each point in the code path that could reintroduce a pull toward the mastered cell needs to be addressed. The amendment sequence is the architectural decomposition of "no signal pulling back" into the specific code paths that needed modifying.

The v0.11 paper will report this decomposition explicitly. The finding "bounded mastery requires four architectural interventions, not one" is itself a contribution worth recording.

## 7. Stopping rule for further amendments

If the v0.11.2 implementation produces another implementation-time discovery requiring further architectural change, the next response will not be a v0.11.3 amendment. The v0.11 architectural goal will instead be re-examined at a more fundamental level — whether the per-cell mastery mechanism with depletion is the right architectural approach, or whether bounding the preference system requires a more substantial redesign than incremental amendments can achieve.

This stopping rule is named explicitly here so that the methodological discipline does not erode through repeated amendments. Three amendments within a single iteration is the limit; beyond that, the iteration's design itself is what needs revision, not its implementation details.

## 8. Commitments

Three methodological commitments accompany this amendment:

First, no v0.11.2 code will be written before this amendment document is committed to the public repository.

Second, the amendment specifies a single architectural change: preference accumulation is blocked on mastered cells. Any further implementation-time discovery requiring additional architectural change will trigger the stopping rule in Section 7, not a further amendment.

Third, the v0.11.2 batch results will be reported alongside both the v0.11 probe finding and the v0.11.1 probe finding. The full progression from v0.11 to v0.11.2 is part of the research contribution: the architectural decomposition of "bounded mastery" into the four code paths that needed modification.
